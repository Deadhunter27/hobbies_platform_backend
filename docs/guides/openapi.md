# OpenAPI Generation Strategy

Governing decision: ADR-0005. The OpenAPI document is the **single source of truth** for the API contract, generated from code.

## Pipeline

```
NestJS controllers + Zod DTOs (ADR-0011)
        │  @nestjs/swagger (+ zod-openapi bridge)
        ▼
openapi/openapi.json          ← committed, diffed in every PR
        │
        ├── Swagger UI at /api/docs (non-production or auth-gated)
        └── consumed by the MOBILE repo → generated TS types/client
```

## Rules

1. **Every endpoint is fully described** — summary, params, request/response schemas, error envelope, auth requirements. An endpoint invisible in the spec fails review.
2. **The spec is regenerated by a script (`pnpm openapi:generate`) and committed.** CI regenerates and fails if the committed file is stale — the contract can never drift from the code.
3. **Contract review happens on the diff.** A PR touching `openapi.json` is a contract change; breaking diffs require a version bump per ADR-0012.
4. **The backend owns the spec; consumers own their codegen.** The mobile repo pins a spec version and generates types with its own tooling — no hand-written duplicate contracts anywhere (cross-repo agreement).
5. Error responses reference the **single error envelope** from the API conventions, with documented `code` values per endpoint.
