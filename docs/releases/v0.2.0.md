# v0.2.0 — Identity & Access

**Date:** 2026-07-16 · **Commits:** `4be0bb6..a5e648b` · **CI:** [run 29456857611](https://github.com/Deadhunter27/hobbies_platform_backend/actions/runs/29456857611) ✅ · **CodeQL:** [run 29456857621](https://github.com/Deadhunter27/hobbies_platform_backend/actions/runs/29456857621) ✅

## Highlights

- **Full authentication stack**: registration, login, token refresh, logout, and password change — argon2id password hashing, 15-minute HS256 access JWTs, and 256-bit opaque refresh tokens stored only as SHA-256 hashes.
- **Rotation with theft detection**: every refresh rotates the token within a session *family*; replaying a rotated token revokes the entire family (including the legitimate successor) and writes an audit entry.
- **Default-deny authorization layer**: a single `can(actor, action, resource)` entry point with a relationship-scoped grants table, ready for M4's community roles without interface changes. Roles are never embedded in tokens — suspension takes effect on the next request, not at token expiry.
- **Append-only audit trail**: every auth-lifecycle event is recorded in the same database transaction as the change it describes, carrying the request correlation id; failed logins store an email *hash*, never the address.

## New endpoints

| Method | Path                           | Auth   | Notes |
| ------ | ------------------------------ | ------ | ----- |
| POST   | `/api/v1/auth/register`        | public | 201; `EMAIL_ALREADY_REGISTERED`, `PASSWORD_TOO_WEAK` |
| POST   | `/api/v1/auth/login`           | public | timing-safe; `INVALID_CREDENTIALS` never reveals account existence |
| POST   | `/api/v1/auth/refresh`         | public | rotates; `SESSION_REVOKED` on reuse detection |
| POST   | `/api/v1/auth/logout`          | bearer | 204, idempotent, revokes the presenting session |
| POST   | `/api/v1/auth/change-password` | bearer | 204, revokes **all** sessions |
| GET    | `/api/v1/me`                   | bearer | authenticated identity echo |

New stable error codes: `EMAIL_ALREADY_REGISTERED`, `INVALID_CREDENTIALS`, `INVALID_REFRESH_TOKEN`, `SESSION_REVOKED`, `USER_SUSPENDED`, `PASSWORD_TOO_WEAK`. The OpenAPI spec now declares a bearer security scheme; protected operations are marked.

## Database changes

One additive migration (`20260715222641_m2_identity_access_audit`):

- `identity_user` — ULID PK, case-insensitively unique (normalized) email, argon2id hash, `status active|suspended|deleted`, `globalRole user|staff`
- `identity_session` — separate aggregate: unique `refreshTokenHash`, rotation `familyId`, `revokedAt`/`replacedById`, device label
- `access_resource_role` — grants with unique `(userId, resourceType, resourceId)`; platform scope by convention
- `audit_entry` — append-only, JSONB metadata, `requestId`, indexed `(resourceType, resourceId, createdAt)`

Applies cleanly from zero on top of the M1 migration (verified fresh-database proof).

## Security improvements

- Login verifies a real argon2id dummy hash for unknown emails — response timing does not reveal account existence (unit-proven).
- Refresh tokens never exist in plaintext at rest or in logs; Pino redaction extended to `accessToken`, `currentPassword`, `newPassword`, `passwordHash`.
- JWT payload is exactly `sub`, `sid`, `iat`, `exp` — verified in e2e by decoding the token.
- `JWT_SECRET` is fail-closed configuration (min 32 chars; boot refuses without it).

## Architecture decisions

- [ADR-0017](../adr/0017-authentication-tokens-and-sessions.md) — short-lived JWT + rotating opaque refresh tokens, argon2id, session families
- [ADR-0018](../adr/0018-authorization-policy-layer.md) — relationship-scoped policy layer, default deny, port-inverted guard (access imports no other module)
- [ADR-0019](../adr/0019-audit-trail.md) — append-only audit trail written in-transaction via a shared UnitOfWork port

## Breaking changes

None. All changes are additive; M1's public catalog and health surfaces are byte-compatible (e2e-verified).

## Migration notes

1. Set `JWT_SECRET` (≥32 chars) in every environment **before** deploying — the app will not boot without it. Optional: `ACCESS_TOKEN_TTL_SECONDS` (default 900), `REFRESH_TOKEN_TTL_DAYS` (default 14).
2. Run `pnpm db:migrate` (`prisma migrate deploy`) — additive only, no data backfill required.

## Known limitations

- No rate limiting on auth endpoints yet (Redis wiring pending; tracked in `docs/roadmap.md`) — do not expose to hostile traffic before it lands.
- No email verification flow (no ADR yet; registration is trusted input).
- Route protection is opt-in via `@RequiresAuth()` (documented posture in ADR-0018).
- `EMAIL_ALREADY_REGISTERED` is a deliberate, documented enumeration trade-off.
- No error tracker wired yet (blocked on deployment-target decision).
